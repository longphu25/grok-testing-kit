---START---

Applying the **Risk-Based Testing (RBT)** strategy, generate a detailed Test Case list from the High-Level Scenarios reviewed in Step 4.

## Requirements for each Test Case:

### 1. Risk Assessment

Rate **Risk Level** for each Module/Feature:
- **High Risk:** Core business, related to money/security, many dependent users → Create many test cases, test thoroughly
- **Medium Risk:** Important but not critical features → Moderate testing
- **Low Risk:** Secondary features, little impact → Basic test, happy path

### 2. Test Case Structure

Each Test Case must have complete:

| Field | Request |
|-------|--------|
| **Module / Sub-module** | Module name decomposed in Step 3 |
| **Test Case Title** | Short title, clear purpose |
| **Pre-conditions** | Prerequisites before testing |
| **Test Steps** | Steps, **numbered in order** |
| **Expected Results** | Expected results correspond to each step, **numbered** |
| **Test Data** | Specific data (see rules below) |
| **Priority** | High / Medium / Low |
| **Risk Level** | High / Medium / Low |

### 3. Test Data Rules

Test Data **MUST BE SPECIFIC**, do not use general descriptions:

```
❌ WRONG: "Enter a valid email"
✅ CORRECT: "Enter email: test_customer_01@domain.com"

❌ WRONG: "Enter a valid code"
✅ CORRECT: "Enter code: KH-2026-0012"

❌ WRONG: "Enter the phone number in the wrong format"
✅ CORRECT: "Enter phone number: abc123xyz (letters instead of numbers)"

❌ WRONG: "Enter name is too long"
✅ CORRECT: "Enter name: [string of 256 consecutive 'A' characters] (255 limit exceeded)"
```

### 4. Coverage

Ensure Test Cases cover a variety of:
- **Happy Path:** Main flow goes smoothly
- **Negative Path:** Incorrect input, missing data, wrong format
- **Boundary Values:** Boundary values (min, max, min-1, max+1)
- **Edge Cases:** Timeout, lost connection, concurrent access

[Additional suggestions (customized by project):
- Decentralization: User role A has what rights, what rights it does not have
- Compatible: Responsive design, multi-browser
- Performance: Handle large lists (1000+ records)
- ...]

### 5. Field-Level Validation

**REQUIRED:** When the form/UI has input fields, it must:
1. **List all input fields** on the form/UI under test
2. **Generate separate validation TCs for EACH field** according to the unique characteristics of that field
3. **DO NOT combine** multi-field validation into 1 test case

Apply validation checklist according to field type:

| Field Type | Validation needs testing |
|---|---|
| **Text (Name, Address...)** | Required/Optional · Min length · Max length · Whitespace-only · Special characters (`<>&"'`) · XSS (`<script>alert(1)</script>`) · SQL injection (`' OR 1=1--`) · Unicode/Emoji · Leading/trailing spaces |
| **Email** | Valid format · Missing`@`· Missing domain · Invalid domain · Many`@`· Special characters · Max length · Case sensitivity · Duplicate email (if unique) |
| **Phone** | Index · Valid Prefix (`+84`, `0`) · Min/Max length · Mixed letters · Accents`-`, `.`, space · Invalid area code |
| **Date / DateTime** | Correct format · Date does not exist (`31/02`) · Leap year (`29/02/2024`) · Past/future · Min/Max date · Timezone |
| **Number / Currency** | Min/Max value · Negative numbers · Zeros · Decimal · Non-numeric characters · Overflow · Leading zeros · Currency format |
| **Dropdown / Select** | Default value · All options · Option disabled · Required (unchecked) |
| **Checkbox / Radio** | Default state · Check/Uncheck · Required · Radio group (only 1) |
| **File Upload** | Valid/invalid file type · Max size · Empty file (0 KB) · File name with special characters · Multiple files |
| **Password** | Min/Max length · Special characters · Upper/lowercase letters · Numbers · Copy-paste · Show/hide · Confirm password |
| **Textarea** | Max length · Line breaks · HTML tags · Character counter |

**Example Field-Level Validation for "Register" form:**```
The form has 4 fields: Full name, Email, Phone number, Password

→ Field "Full name" (Text, max 100 characters, required):
  - TC: Leave blank → Report required error
  - TC: Enter 1 character → Valid (or report min length error)
  - TC: Enter 100 characters → Valid (max)
  - TC: Enter 101 characters → Report error exceeding max
  - TC: Enter only spaces → Report error
  - TC: Enter <script>alert(1)</script> → Do not execute script

→ "Email" field (Email, required, unique):
  - TC: Enter test@domain.com → Valid
  - TC: Enter testdomain.com (missing @) → Report format error
  - TC: Enter test@.com (missing domain) → Report error
  - TC: Enter an existing email → Report duplicate error

→ Field "Phone number" (Phone, 10 numbers, starting with 0):
  - TC: Enter 0912345678 → Valid
  - TC: Enter 912345678 (missing prefix 0) → Report error
  - TC: Enter 091234567 (9 numbers) → Report min length error
  - TC: Enter abc1234567 → Report an error that only accepts numbers

→ "Password" field (Password, 8-20, needs uppercase + number):
  - TC: Enter Abcdef12 (8 characters, upper case + number) → Valid
  - TC: Enter abcdefgh (lowercase only) → Report error
  - TC: Enter 1234567 (7 characters) → Report min length error
```

> Determine the characteristics of each field (type, required, min/max, format, unique...) from requirements before generating TCs.

### 6. Test Case Design Techniques (Test Design Techniques)

Apply the following classic techniques to create systematic test cases:

**a) Equivalence Partitioning**
Divide input data into equivalent groups, only need to test 1 representative value in each group.```
Example with "Age" field (valid: 18-60):
- Eligible class: 25 (representatives 18-60)
- Invalid class below: 10 (representation <18)
- Invalid class above: 70 (representation >60)
```

**b) Boundary Value Analysis — BVA (Boundary Value Analysis)**
Test values at the boundary: min, min+1, max-1, max, min-1, max+1.```
Example with "Password" field (6-20 characters):
- 5 characters (lower border -1) → Report error
- 6 characters (lower border) → Valid
- 7 characters (lower border +1) → Valid
- 19 characters (top boundary -1) → Valid
- 20 characters (upper border) → Valid
- 21 characters (upper boundary +1) → Report error
```

**c) Decision Table**
Used when logic has multiple combination conditions (AND/OR). List the combination of conditions → results.```
Example with "Login" function:
| Valid email | Correct password | Active account | Results |
|:---:|:---:|:---:|---|
| ✅ | ✅ | ✅ | Log in successfully |
| ✅ | ❌ | ✅ | Report wrong password |
| ❌ | ✅ | ✅ | Report email does not exist |
| ✅ | ✅ | ❌ | Report account locked |
```

**d) State Transition**
Used for objects with many transition states (workflow, order status...).```
Example with Order status:
Newly created → Processing → Delivering → Completed
                      → Cancel order
Test: Each valid transition + NOT valid transitions (eg: Completed → Newly created)
```

> Choose the appropriate technique for each Module/Field based on data characteristics and business logic.
