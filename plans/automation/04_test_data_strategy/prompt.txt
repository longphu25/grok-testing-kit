Test automation becomes flaky if using hardcode for unique fields (Email, Username, Customer Code...).

Please design **Test Data Strategy** according to the following requirements:

## 1. Data Generator class

Write a Utils/Helper class to automatically generate data:
- Use the appropriate library: **Faker** (JS/TS), **JavaFaker** (Java), or UUID
- Each function generates a specific type of data

## 2. Format traceable (traceable)

Data must follow the format:```
[Prefix]_[TestName]_[Timestamp]_[Random]
```

Specific examples:```
Email:    auto_createCustomer_20260402_A3F2@test.com
Username: auto_login_20260402_7B4D
Code:     KH-20260402-A3F2
Phone:    09[random 8 digits]
```

> Look at the database or log → immediately know which test this data was generated by.

## 3. Data for each type of test

| Test Type | Request Data | Example |
|-----------|-------------|-------|
| **Happy Path** | Valid data, correct format |`valid_email@test.com`, `Test@12345`|
| **Negative** | Data is in wrong format, missing field |`invalid-email`, `""`, `null`|
| **Boundary** | Data at the min/max boundary |`"A"` (1 char), `"A"*256`(exceeds max) |

## 4. Integrate into project

Instructions on how to use this data set in Test Cases:

Playwright:
```typescript
import { DataGenerator } from '../utils/data-generator';
const testData = DataGenerator.createCustomer();
```

Selenium/Java:
```java
import utils.DataGenerator;
String email = DataGenerator.generateEmail("createCustomer");
```

## 5. Mandatory rules (ref`.agents/rules/automation_rules.md`)

- ❌ DO NOT hardcode data for unique fields
- ❌ DO NOT use duplicate data between tests
- ✅ Data must be unique each time it runs
- ✅ Data must be traceable (know which test is generated)
- ✅ Support running parallel without conflict
