# API SPECIFICATION

## Method 1: Swagger/OpenAPI URL
- Swagger URL: [https://api.example.com/swagger.json]
- (Supported: Swagger UI, Scalar, Redoc, Stoplight, RapiDoc — Agent automatically finds the original URL spec)

## Method 2: Manual API description
- Base URL: [https://api.example.com/v1]

POST /api/customers
- Request: { name, email, phone, age }
- Response 201: { id, name, email, phone, age, createdAt }
- Response 400: { error: "Validation failed" }

GET /api/customers
- Query params: page, limit, sort, search
- Response 200: { data: [...], total, page, limit }

GET /api/customers/{id}
- Response 200: { id, name, email, phone }
- Response 404: { error: "Not found" }

PUT /api/customers/{id}
- Request: { name, email, phone }
- Response 200: { id, name, email, phone, updatedAt }

PATCH /api/customers/{id}
- Request: { name } (partial update)
- Response 200: { id, name, email, phone, updatedAt }

DELETE /api/customers/{id}
- Response 204: No content
- Response 404: { error: "Not found" }

# OUTPUT FORMAT
## Mode SPEC:
1. API Overview (base URL, auth, endpoints)
2. Detailed Test Cases according to 7 types:
   - Happy Path
   - Negative (Validation + Auth)
   - Boundary Values
   - Edge Cases
   - Field-Level Validation (each field in the request body)
   - Security (injection, IDOR, sensitive data)
   - Pagination & Filtering (for GET list)
3. Test Data Matrix (valid / invalid / boundary)
4. PUT vs PATCH distinction (if any)
5. Execution Order & Dependencies

## FULL Mode:
1. All Mode SPEC, plus:
2. API Client/Helper class (base URL, auth, logging)
3. Model/DTO classes
4. Test classes (assertions: status code + body + schema + headers)
5. Test Data generators (unique + traceable)
6. Test results (PASS/FAIL summary)

# NOTES
- Mode SPEC: Agent only generates test cases in document form
- FULL Mode: Agent generates code + runs test + auto-heal if FAIL
- Agent is REQUIRED to list each field in the request body and generate separate validation TCs for EACH field
- Agent must generate Security test cases (injection, IDOR, auth bypass, sensitive data)
